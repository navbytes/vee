# Vee — Building a Native macOS Launcher with a JavaScriptCore Plugin Platform: A Deep Implementation Report

*Vee is a keyboard-first, plugin-extensible macOS launcher and menubar platform — a fast, local, beautiful successor to Hammerspoon. This document is the implementation spec for Vee's engine and its initial plugins.*

## TL;DR
- **Run plugins out-of-process and render declaratively, exactly as Raycast does.** Raycast's own engineering blog confirms their production architecture: a managed external runtime hosting isolated JS engines, JSON-RPC over stdio file descriptors (via Swift's DispatchIO), a custom React reconciler that emits a JSON "render tree" diffed with JSON Patch, and native AppKit view models on the receiving side. Vee should copy this pattern; it is battle-tested.
- **Split fetch from filter at the IPC boundary.** Push the candidate set from plugin to host once per open/refresh, then run a native fzy/fzf-style fuzzy matcher per keystroke entirely in Swift — never cross the process boundary on keypress. Use stale-while-revalidate (RFC 5861) so the cached set renders instantly while a background refresh runs.
- **Lean on proven open source for each plugin.** Maccy (clipboard, NSPasteboard changeCount polling + concealed-type exclusion), MeetingBar (EventKit + regex meeting-link detection), KeyboardShortcuts by Sindre Sorhus (Carbon RegisterEventHotKey global hotkeys), and the GitHub/Jira REST+GraphQL APIs with tokens in the macOS Keychain.

---

## Key Findings

1. **JavaScriptCore is the right embedding engine for Vee, but it is bare.** JSC ships no `fetch`, no timers, no `console`, no module loader. Vee must bridge all of these natively. Retain cycles between `JSContext` and exported native objects are the central memory hazard; `JSManagedValue` plus a careful "never capture context in a block" discipline is the fix.
2. **Raycast tried JSC-per-extension first, then moved to Node+workers — but for Vee, JSC is the better choice.** Raycast abandoned pure JSC because developers demanded the full npm ecosystem (OctoKit etc.). For a single-user/small-team tool where you author the plugins, JSC's "sandbox for free" and zero-runtime-download properties are advantages, not liabilities.
3. **The declarative renderer is the hardest component, and `react-reconciler` is the proven path.** A custom host config turns a React tree into host instances; you serialize that tree to JSON, diff against the previous render with JSON Patch, and ship only the patches across IPC.
4. **Keystroke latency is an architecture problem, not an optimization.** The only robust answer is to keep the candidate dataset host-side and filter natively. Raycast exposes exactly this knob: `filtering={false}` / `onSearchTextChange` hands filtering to the extension; the default keeps it native.
5. **Clipboard managers have no event API — they poll `changeCount` — and the privacy model is a hard requirement, not a nicety.** Honoring `org.nspasteboard.ConcealedType`/`TransientType`/`AutoGeneratedType` and password-manager UTIs is the difference between a safe tool and one that logs passwords to disk.
6. **EventKit on macOS 14+ has a bifurcated permission model** (`requestFullAccessToEvents` + `NSCalendarsFullAccessUsageDescription` vs. write-only), and reading events requires full access.

## PART 1 — THE VEE PLUGIN HOST ENGINE

### 1. JavaScriptCore embedding & the native bridge

**Core API surface.** The `JavaScriptCore` framework exposes four classes and one protocol: `JSVirtualMachine` (a self-contained heap/GC world), `JSContext` (an execution environment bound to one VM), `JSValue` (a wrapped JS value), `JSManagedValue` (conditional-retain wrapper for breaking cycles), and the `JSExport` protocol. Contexts within the same VM can share objects; contexts in different VMs cannot communicate. JS execution within a single VM is serialized — "any attempts to access code running in a JSVirtualMachine from a separate thread will wait until the initial thread is finished" — so concurrency requires multiple VMs on multiple threads.

**Evaluating and calling across the boundary:**
- Swift→JS evaluate: `context.evaluateScript("1+2+3")` returns a `JSValue`; cast with `.toInt32()`, `.toString()`, `.toDictionary()`, etc.
- Swift→JS call: `context.objectForKeyedSubscript("fnName")` then `.call(withArguments:)`.
- JS→Swift via blocks: assign a `@convention(block)` closure into the context: `context.setObject(unsafeBitCast(block, to: AnyObject.self), forKeyedSubscript: "name")`. This is how `console.log` is typically bridged.
- JS→Swift via objects: declare an `@objc` protocol inheriting `JSExport`, implement it on an `NSObject` subclass, and `setObject` an instance. Only members declared in the JSExport protocol are exposed. Swift initializers are not auto-exported; expose a factory via a `@convention(block)` cast of `.init`.

**Exception handling.** `evaluateScript(_:)` does not throw and fails silently on syntax/runtime errors. You must set `context.exceptionHandler = { context, exception in ... }` before evaluating. A common pattern is to rethrow native errors into JS with `JSContext.current().evaluateScript("throw ...")` and to capture JS exceptions as native `Error`s so plugin authors get real stack traces.

**Memory management — the critical pitfall.** "You will create a retain cycle anytime you export a Swift/Objective-C object to JavaScript and store the value… A JSContext maintains a strong reference to any Swift/Objective-C object you pass to JavaScript." Two rules:
- **Never capture `context` inside an exported block.** NSHipster: "make sure not to reference context within your closures; instead, you can access it by way of the `JSContext.current` type property."
- **Use `JSManagedValue` for any `JSValue` you store on a native object** (e.g., a stored JS callback). `JSManagedValue` provides "conditional retain behavior" — add it to the VM with `JSVirtualMachine.addManagedReference(_:withOwner:)` so the value lives as long as the owner without forming a cycle.

**What Vee must polyfill/bridge** (JSC has none of these):
- **`console.log`/`warn`/`error`** → bridge to native logging (Raycast streams these via an OS log stream captured by its CLI).
- **`setTimeout`/`setInterval`/`clearTimeout`** → back with native `DispatchSourceTimer` or `Timer`; maintain an ID→timer map on the native side and a JS callback registry. **Pitfall (Apple Developer Forums):** JSC has no built-in event loop, so a native-triggered timer callback can run *between* the microtask queue and the next macrotask in a way that differs from Node/Safari ordering. The forum poster's fix: "set a flag on the JavaScript side indicating a native call is being made, and if any native-triggered callback occurs… when this flag is set, I have to 'queue' it via a promise handler for execution AFTER the current event loop is complete." Plan for a real microtask/macrotask queue if you want Node-compatible Promise ordering.
- **`fetch`/`XMLHttpRequest`** → implement natively with `URLSession` and resolve a JS `Promise`. AppCoda confirms "we cannot make HTTP requests from JavaScript using the JavaScriptCore framework." Expose an async native function that returns a promise the bridge resolves on completion.
- **Module loading / `require`/`import`** → JSC has no loader; bundle plugin TS/JS into a single file with esbuild (below) so there are no runtime imports.

**Curated API surface design.** Vee should expose a small, namespaced, capability-gated global (e.g., `vee.http.fetch`, `vee.fs.readScoped`, `vee.clipboard`, `vee.storage`, `vee.ui`, `vee.keychain`). Two reference wrappers worth studying: **ELJSBridge** (Electrode-iOS, now deprecated but illustrative) wraps the JSC ObjC bridge with an `addExport(_, name:)` pattern and re-adds exports when the context is swapped — exactly what Vee needs for hot reload. Ilya Puchka's "Extending native code with JavaScriptCore" documents the adapter-class technique for keeping your core types free of JSC references and rethrowing JS exceptions as native errors.

### 2. Out-of-process plugin execution

**Why out-of-process.** Crash isolation: "If an XPC service crashes, it won't bring down your main application." Raycast's blog states they deliberately run extensions in a child process so "the Node process can crash without crashing Raycast itself (most of the time)." With their worker-isolation model: "A worker can crash or run out of memory; we catch this and show an error screen for the extension but Raycast itself is largely unimpressed by that." Vee should adopt the same stance: a misbehaving plugin shows an error surface, the launcher stays up.

**Two viable macOS architectures:**

**(A) XPC service (`NSXPCConnection`).** This was Raycast's *first* attempt — verbatim: "And that's what we did: one Raycast main process, one XPC support process (the extension host) and many JavaScriptCore engines running in that support process. We hacked a proof-of-concept and it all worked nicely… suspicious." For Vee's JSC-based design this is the natural fit because you are not shipping Node.

- Define an `@objc` protocol; all methods must return `void` (XPC is asynchronous) and use reply blocks for results. Set `exportedInterface`/`exportedObject` and obtain `remoteObjectProxyWithErrorHandler`.
- **Crash/hang supervision is built in.** Implement both handlers: the **interruption handler** "is called when the remote process crashes but the local connection object is still valid, allowing you to attempt reconnection," and the **invalidation handler** is "called when the invalidate method is called or when an XPC helper could not be started." launchd respawns the service on the next call. Design the service to be near-stateless: "Because an XPC service can be terminated suddenly at any time, it must be designed to hold on to minimal state."
- Serialization: NSXPC uses `NSSecureCoding`; pass only arithmetic types, strings, C structs of those, or `NSSecureCoding` objects. Validate the peer with `SecCodeCreateWithXPCMessage` against a `SecRequirement` to prevent spoofed connections.
- **Pitfall:** an XPC service's main thread calls `dispatch_main`, which "terminates the main thread but leaves the process running." If you call AppKit (which demands the main thread) inside the service you'll crash; keep the JS host headless and do UI only in the main app.

**(B) Plain child process + Unix-domain-socket/stdio JSON-RPC.** This is Raycast's *shipping* design: "streams on the standard file descriptors (in Swift via the DispatchIO framework)" carrying **JSON-RPC**, chosen because it is "straightforward and lightweight," OSS-supported (their native impl is "based on a part of Apple's language server implementation"), "fast enough," and "easy to debug due to its plain text nature." For low-latency, you can swap the JSON-RPC payload for MessagePack, but Raycast found plain JSON + selective gzip ("a threshold where gzip compression would save us more time than the actual compress/uncompress step takes") sufficient. The CodeEdit project independently arrived at the same conclusion for its extension host: "The architecture involves using node and communicating with the CodeEdit application using JSON-RPC. A react-reconciler will allow extensions to create a custom UI with React components."

**Per-plugin process vs. shared host.** Raycast rejected one-process-per-extension as "expensive" and instead runs **one host process with per-extension isolation via V8 worker threads** (V8 isolates), with per-worker heap memory limits — "extensions that get too greedy will be stopped without a heads-up." For Vee's JSC design the analogue is **one host process containing one `JSContext` per plugin on a shared (or per-plugin) `JSVirtualMachine`** — separate VMs give you separate GC heaps and the ability to tear one down without touching others. Recommendation: **shared host process, one JSContext-per-plugin, separate JSVirtualMachine per plugin** for true heap isolation, accepting the higher memory cost only if a plugin needs concurrent execution.

**Ordering/concurrency.** Raycast: "In Swift, we use serial queues and buffered streams to make sure messages arrive and leave in order." Use a serial `DispatchQueue` per connection for decode/dispatch, hop to a background queue for work, and to the main queue only for UI. Race conditions *between* processes still need custom logic.

### 3. The declarative UI renderer (the hard part)

**The proven design (Raycast, verbatim mechanism):** developers write React using host-provided components (`<List>`, `<Detail>`, `<Form>`, `<Action>`). A **custom React reconciler** "converts your React component tree to a render tree." After each render pass they "create a JSON representation for each component… composing in the reconciler to what we call the 'render tree'." They diff successive trees with **JSON Patch** (RFC 6902): "If there are no patches, it means there are no changes. So native land can just sit there and do nothing." Patches are sent over IPC, where Swift builds "lightweight Swift view models and translate[s] patches to bitset types," then "pass[es] it all down to custom AppKit components." This view-model layer "shield[s] native components from changes of the render tree representation."

**How Vee should build the reconciler with `react-reconciler`:** Install the experimental `react-reconciler` npm package and supply a **host config** object. The key methods (from the React README and the nitin42 / atulr tutorials, and Sophie Alpert's ReactConf 2019 talk):
- `createInstance(type, props, rootContainer, hostContext)` → create your node descriptor (e.g., `{type, props, children:[]}`), not an AppKit view (that lives across the boundary).
- `createTextInstance`, `appendInitialChild`, `appendChild`, `appendChildToContainer`, `insertBefore`, `removeChild` → mutate your in-memory tree.
- `prepareUpdate(instance, type, oldProps, newProps)` → compute a prop diff; `commitUpdate` applies it.
- `finalizeInitialChildren` → return `true` to get a `commitMount` callback (e.g., autofocus).
- `getRootHostContext`/`getChildHostContext` → thread context (analogous to DOM-vs-SVG; here e.g. List-vs-Form context).
- `prepareForCommit`/`resetAfterCommit` → in `resetAfterCommit` you serialize the new tree and emit the JSON-Patch diff to native.
- `now`/scheduling → use `Date.now`/`performance.now`.
You wrap the reconciler with a public `render(element, container)` that calls `Reconciler.createContainer` then `updateContainer`.

**The diffing strategy to avoid resending whole trees:** keep the previous serialized tree in the worker; after each commit compute `jsonpatch.compare(prev, next)` and send only the patch array (RFC 6902 ops: add/remove/replace/move). Native applies the patch to its mirror of the tree and updates only the affected view models. This is strictly better than React Native-style per-node mutation messages when crossing a process boundary because it batches and naturally coalesces.

**Simpler alternative (recommended to start Vee with):** you do **not** need React. Define a plain declarative protocol: the plugin returns a JSON component tree on each update (`{type:"list", children:[{type:"list-item", props:{title, subtitle, icon, actions}}]}`). Diff with JSON Patch the same way. You can later add React purely as an authoring convenience on top of the same wire format. Project Gauntlet and the ByteAtATime "raycast-linux" project both reimplement this Raycast-style reconciler-to-JSON pipeline and are readable references; CodeEdit's extension-architecture discussion documents the same JSON-RPC + react-reconciler decision.

### 4. Hot reload

**File watching.** Use **FSEvents** (coarse, directory-level, low overhead) or a `DispatchSource` file-descriptor source (`DispatchSource.makeFileSystemObjectSource`) for specific files. On change, trigger a rebuild.

**Transpile/bundle on the fly with esbuild.** Raycast's `ray` CLI integrates "esbuild in its library version to transpile TypeScript code into a JavaScript bundle." Use esbuild's **context API** (`esbuild.context(...)` then `ctx.watch()`), which does incremental rebuilds: "subsequent builds are done incrementally (i.e. they reuse some work from previous builds)." esbuild caches file contents and parsed ASTs in memory between rebuilds. Enable `sourcemap: true` for debugging; JSC honors source maps for stack traces. (swc is a faster-still alternative but esbuild's ecosystem and Raycast precedent favor it.)

**Re-evaluating in a fresh context.** On rebuild: tear down the plugin's `JSContext` (and optionally its `JSVirtualMachine`), create a fresh one, re-install the curated API globals and polyfills, and re-evaluate the new bundle. This is the clean-state model. To **preserve state** across reloads, serialize the plugin's `vee.storage` (and React state if desired) to a plain JSON blob before teardown and rehydrate after — never try to carry live `JSValue`s across a context swap. ELJSBridge's pattern of re-adding all exports when `context` is reassigned is the template.

**Dev-loop wiring.** Raycast's CLI talks to the app over a custom URL scheme (`raycast://`) and a pid file rather than running a server in-app. A simpler local design for Vee: the file watcher lives in the host; on esbuild's `onEnd` callback (the post-0.17 replacement for `onRebuild`) it signals the host to reload the affected plugin's context.

### 5. The fetch-vs-filter latency split

**Principle:** crossing the IPC boundary on every keystroke is fatal to perceived latency. Push the candidate set **once** per open/refresh; filter **natively** per keystroke.

**Raycast's exact knob:** `<List>` has a `filtering` prop — "When true, Raycast will use the query in the search bar to filter the items. When false, the extension needs to take care of the filtering." Filtering is *implicitly* false if you set an `onSearchTextChange` listener. Their native filtering "uses built-in filtering by indexing the title of list items and additionally keywords." For Vee: the plugin sends items (each with `title`, `subtitle`, `keywords`, `id`, `actions`); the **host** owns the search field and runs the matcher; `onSearchTextChange` is only fired across IPC when the plugin explicitly opts into server-side filtering (e.g., a remote search API), and Raycast recommends `throttle: true` in that case.

**The native matcher.** Implement an fzy-style scorer in Swift. fzy "treats scoring as a modified edit distance problem" using a dynamic-programming algorithm "similar to Wagner–Fischer and Needleman–Wunsch" with an **affine gap penalty** (constant penalty for a gap, linear for its length), favoring "matches on consecutive letters and starts of words." fzf's `FuzzyMatchV2` is the other canonical reference (with ASCII fast-path and slab memory reuse to "minimize garbage collection"). For Swift you can port fzy's algorithm directly or call into a Rust matcher like **Frizbee** (SIMD Smith-Waterman, "outperforms nucleo by ~1.8x and fzf by ~2.3x").

**Data structures & sync.** Hold candidates in a contiguous Swift array of value-type structs (cache-friendly); precompute a lowercased match string and word-boundary bitmasks per item at ingest time so per-keystroke work is pure scoring. For large sets, process in batches and yield to the main thread to avoid blocking input rendering (the elanmed fuzzy-finder writeup documents UI stalls from doing all scoring synchronously and fixes them with batching). For **incremental updates** (SWR revalidation returning a changed set), diff by stable `id` and patch the array in place rather than rebuilding, preserving the current selection. Raycast's `useStreamJSON` exists precisely for candidate sets too large to fit in memory — it streams a JSON resource from disk and filters with a `filter` callback; mirror this for very large datasets.

### 6. Stale-while-revalidate caching

**Conceptual model (RFC 5861, popularized by Vercel SWR):** "SWR first returns the data from cache (stale), then sends the request (revalidate), and finally comes with the up-to-date data again." Raycast's `useCachedPromise`/`useFetch` implement exactly this: "first returns the data from cache (stale), then executes the promise (revalidate), and finally comes with the up-to-date data again."

**Implementation pattern for Vee:**
1. On plugin open, synchronously read the last-cached payload and render immediately.
2. Kick off the async refresh (network/disk) in the background.
3. On success, atomically swap the cache and re-render (emit a new render tree → JSON Patch → native update).
4. Use `keepPreviousData: true` semantics to avoid flicker when arguments change: Raycast notes the default behavior "can cause some flickering (initial data → fetched data → arguments change → initial data → fetched data)" and `keepPreviousData` fixes it.

**Mutation/optimistic updates:** expose a `mutate` that wraps an async update; "by default, the data will be revalidated… after the update is done," with an `optimisticUpdate` function and automatic `rollbackOnError`.

**Pluggable caching API for plugin authors.** Model the interface on the `stale-while-revalidate-cache` npm package: a `Storage` with `getItem`/`setItem`/`removeItem`, plus `minTimeToStale` (when to start considering data stale) and `maxTimeToLive` (hard expiry), `serialize`/`deserialize` hooks, and an event emitter for cache hit/miss/expired/revalidate. Expose to Vee plugins as:

```ts
interface CacheStrategy<T> {
  get(key: string): T | undefined;
  set(key: string, value: T, opts?: {ttl?: number}): void;
  invalidate(key: string): void;
}
```

Ship a default disk-backed strategy (per-command support folder, like Raycast's Cache namespace) and an in-memory LRU, and let a plugin supply its own object implementing the protocol. TTL, manual invalidation, and persistent-vs-memory are all just configuration of the same interface.

## PART 2 — IMPLEMENTATION RECIPES FOR VEE'S SIX LAUNCH PLUGINS

### Plugin 1 — Clipboard history (persisted, searchable, configurable count)

**Monitoring (no notification API exists).** macOS provides no clipboard-change event; the universal technique is to **poll `NSPasteboard.general.changeCount`** on a timer. Maccy defaults to a 500 ms interval and exposes it as `clipboardCheckInterval`, tunable down to 100 ms (documented verbatim in the README: `defaults write org.p0deje.Maccy clipboardCheckInterval 0.1 # 100 ms`). When `changeCount` increments, read the new contents.

**Capturing multiple representations.** Maccy's `Clipboard.swift` declares its `supportedTypes` set as `.fileURL, .html, .png, .rtf, .string, .tiff`, plus handling of dynamic types (`dyn.` prefix) and Microsoft Office source types (`com.microsoft.ole.source.`). Read all available types per pasteboard item so paste fidelity is preserved.

**The privacy requirement (mandatory for Vee).** Honor the NSPasteboard.org conventions. Maccy's `ignoredTypes` set is `.autoGenerated, .concealed, .transient`. Per nspasteboard.org:
- **`org.nspasteboard.ConcealedType`** — "content that should be treated as confidential… Avoid recording it to a file, or consider encrypting it." Set by password managers.
- **`org.nspasteboard.TransientType`** — "content will be on the pasteboard only momentarily… Data marked transient should not be recorded or displayed in a pasteboard history."
- **`org.nspasteboard.AutoGeneratedType`** — generated without user intent; "probably should not be recorded as part of the pasteboard history."
- Additionally exclude password-manager proprietary UTIs that Maccy ships by default: **`com.agilebits.onepassword`**, `com.typeit4me.clipping`, `de.petermaurer.TransientPasteboardType`, `Pasteboard generator type`, `net.antelle.keeweb`. Maccy's site confirms it "honours the concealed flag set by 1Password and Bitwarden, so secrets are dropped automatically." Provide a user-editable blocklist (use the free Pasteboard-Viewer app to discover an app's custom types) and a temporary "ignore next copy" toggle.

**Persistent storage.** Persist to a local store (Maccy uses Core Data; SQLite via GRDB is an equally good choice and gives you FTS). Apply **file protection** and consider encryption for the on-disk store (see cross-cutting section). Configurable item count is a simple "history size" preference that trims oldest non-pinned entries — Maccy, for reference, stores up to 999 items configurable, with search staying instant regardless of history size.

**Search/indexing.** Maccy indexes copied text and supports fuzzy search; for SQLite, use an FTS5 virtual table over the text representation, or reuse Vee's native fuzzy matcher (Part 1.5) over the in-memory recent set. Integration into Vee's architecture: the clipboard monitor is a **host-side native service** (it needs `NSPasteboard` + a steady timer that must outlive any single plugin), exposing `vee.clipboard.history(query)` and `vee.clipboard.copy(item)` to plugins. Klipsustreamer is a small open-source reference for the `changeCount` polling loop.

### Plugin 2 — Meeting bar (next meeting in the menu bar)

**EventKit basics.** Use a long-lived `EKEventStore` (Microsoft's docs: "it should be created and destroyed as little as possible… keep that reference around for the entire lifetime of the application").

**Permissions (macOS 14+/iOS 17+ change).** Reading events now requires **full access**: call `requestFullAccessToEvents()` and add **`NSCalendarsFullAccessUsageDescription`** to Info.plist. Write-only access (`requestWriteOnlyAccessToEvents` + `NSCalendarsWriteOnlyAccessUsageDescription`) is insufficient for a meeting bar because you must *read* events. Apple's TN3153 documents the API migration. Check `EKEventStore.authorizationStatus(for: .event) == .fullAccess`.

**Fetching upcoming events.** Build a date-range predicate and query:

```swift
let start = Date.now
let end = start.oneMonthOut  // or next 24h
let predicate = store.predicateForEvents(withStart: start, end: end, calendars: nil)
let events = store.events(matching: predicate).sorted { $0.startDate < $1.startDate }
```

**Meeting-link detection for one-click join.** MeetingBar's `MeetingServices.swift` is the canonical reference: it runs an array of `NSRegularExpression` patterns against the event's location *and notes/description* (a logged bug confirms notes must be searched when the location is a physical room). Example patterns: Zoom, `zoommtg://` native, Teams (`teams\.microsoft\.(com|us)/l/meetup-join/`), Google Meet, Webex, Slack huddle (`app\.slack\.com/huddle/`), Jitsi, etc. — 50+ services. EventKit also exposes structured conference data via `EKEventVirtualConferenceDescriptor` on newer OSes; prefer it when present, fall back to regex.

**Entitlement implication for out-of-process.** A child/XPC process *inherits the sandbox of the parent*, and Raycast notes "if an extension needs access to some items listed in the macOS security preferences, system prompts may appear." Simplest correct design for Vee: **read calendars in the main app (which holds the usage-description entitlement and triggers the TCC prompt)** and expose results to plugins via `vee.calendar.upcoming()`, rather than requesting calendar access from each plugin process. MeetingBar (`leits/MeetingBar`) is the full open-source implementation to study.

### Plugin 3 — API-to-menubar (arbitrary authenticated API, periodic refresh)

**Pattern.** A "no-view"/menu-bar command that fetches on an interval and renders a compact title + dropdown. Raycast shipped exactly this as "menu bar commands" with "Background Refresh… configured with a new interval option in the Extension's manifest."

**Scheduling.** Drive periodic refresh with a native `DispatchSourceTimer` in the host (survives across plugin idle), invoking the plugin's exported refresh function; this is more reliable than a JS `setInterval` whose context may be torn down. Apply **stale-while-revalidate** (Part 1.6): render last value instantly, refresh in background, swap.

**Auth, rate-limiting, offline.** Read the token from Keychain via `vee.keychain` (cross-cutting section). Implement exponential backoff on failure and respect `Retry-After`/rate-limit headers. On offline/error, keep showing the last good cached value with a subtle staleness indicator rather than blanking the menu bar — Raycast's guidance is to "show a toast for all expected errors, e.g. a failing network request" rather than crash.

### Plugin 4 — PRs & Jiras in the launcher (fuzzy list, SWR, own token)

**GitHub — fetching your open PRs.** Two approaches (confirmed against GitHub Docs):
- **REST search:** `GET https://api.github.com/search/issues?q=is:open+is:pr+author:@me` (also `review-requested:@me`, `assignee:@me`; `involves:@me` is the OR of author/assignee/mentions/commenter). Results are under `items`; identify PRs by the `pull_request` key. **Always include `is:pr`** — GitHub App user tokens return HTTP 422 / empty nodes if issues and PRs are mixed. **The Search API rate limit is just 30 requests/minute** for authenticated users (GitHub's `/rate_limit` sample response shows `"search": { "limit": 30 }` vs. `"core": { "limit": 5000 }` and `"graphql": { "limit": 5000 }`) — far lower than the core limit — so cache aggressively and never poll it per keystroke.
- **GraphQL (recommended for a dashboard):** `POST https://api.github.com/graphql` with `search(query: "is:open is:pr author:@me", type: ISSUE, first: 100)` (note `type: ISSUE` returns PRs) or `viewer { pullRequests(states: OPEN, first: 100) }`. GraphQL lets you fetch `number, title, url, isDraft, reviewDecision, repository { nameWithOwner }`, and `statusCheckRollup` **in one round trip**, versus REST search + per-PR follow-ups. GitHub: "one GraphQL call can replace multiple REST calls." The **GraphQL limit is 5,000 points/hour** (min cost 1/call; `first`/`last` must be 1–100; ≤500,000 nodes/call). Include a `rateLimit { cost remaining resetAt }` field to self-monitor.

**GitHub auth.** Prefer a **fine-grained PAT** with **"Pull requests: Read"** (+ "Metadata: Read"); public repos need no extra permission and the search endpoint "does not require any permissions." Classic PATs use the broad **`repo`** scope for private repos. Header: **`Authorization: Bearer <TOKEN>`** (use `Bearer`, not `token`, for fine-grained PATs and JWTs), plus `Accept: application/vnd.github+json` and `X-GitHub-Api-Version: 2022-11-28`.

**GitHub pagination.** REST: `per_page` (max 100; search defaults to 100) + `page`; **follow the `Link` header's `rel="next"`** rather than constructing URLs; the search API caps total results at 1,000. GraphQL: cursor pagination via `pageInfo { hasNextPage endCursor }` → pass `endCursor` as `after`.

**Jira — fetching assigned issues.** Use the Cloud REST endpoint **`/rest/api/3/search/jql`** (⚠️ the legacy `/rest/api/3/search` has been **removed** from Jira Cloud). JQL: `assignee = currentUser() AND statusCategory != Done ORDER BY updated DESC`. Auth is **HTTP Basic with `email:api_token`** (Atlassian API tokens, managed at id.atlassian.com), `Accept: application/json`. POST the JQL (JSON body) to avoid URL-encoding for long queries. **Pagination uses `nextPageToken`** (the old `startAt` is deprecated) — note the community has documented bugs where `nextPageToken` pagination loops/`isLast` never flips, so test pagination carefully and constrain with tight JQL + `maxResults`.

**Combining with SWR + fuzzy filter.** Fetch PRs and Jiras with SWR (render cached list instantly, revalidate in background); push the combined candidate set to the host once; filter per keystroke with the native matcher (Part 1.5). This is the canonical "fetch once, filter natively" flow.

### Plugin 5 — App search (brief)

**Enumeration.** Scan `/Applications`, `/System/Applications`, `~/Applications` for `.app` bundles (and query Launch Services/Spotlight for completeness).
**Launch.** Use `NSWorkspace.shared.urlForApplication(withBundleIdentifier:)` to resolve, then `NSWorkspace.shared.openApplication(at:configuration:completionHandler:)` (the modern replacement for the deprecated `launchApplication`). Note sandboxed apps face restrictions launching by path/URL; the bundle-identifier path is the sandbox-friendly one.
**Ranking & icons.** Apply **frecency** (frequency × recency with exponential decay — e.g., a half-life of ~30 days, score 1 decaying to 0.5 in 30 days, as in the `fre`/fzf-lua-frecency model) combined with the fuzzy score (a weighted sum like `0.7*fuzzy + 0.3*frecency`). Raycast's "Root Search" describes the same behavior: "The more often, and the more recently, you pick a result… the higher it ranks." Cache `NSImage` icons lazily (load on first display, keep an LRU).
**Integration.** App search is a **host-native provider** feeding the same candidate→native-filter pipeline; expose it as a built-in rather than a JS plugin since it needs no network and benefits from native speed.

### Plugin 6 — Per-app global hotkeys (Raycast-style)

**The standard API.** Global hotkeys still require **Carbon `RegisterEventHotKey`** — "Most of the Carbon APIs were deprecated years ago, but there are some left that Apple never shipped modern replacements for. This includes registering global keyboard shortcuts."

**Use a maintained wrapper.** **`KeyboardShortcuts` by Sindre Sorhus** is the recommended library: Swift-native API, a SwiftUI/Cocoa `Recorder` component for user-chosen shortcuts, stores the binding in `UserDefaults`, warns on conflicts, and per the README "is fully sandboxed and Mac App Store compatible. And it's used in production by Dato, Jiffy, Plash, and Lungo." API shape: declare `KeyboardShortcuts.Name`, drop a `KeyboardShortcuts.Recorder` in settings, and register `KeyboardShortcuts.onKeyUp(for:)`/`onKeyDown`. The older **`HotKey`** library is fine for hard-coded shortcuts but "doesn't provide any UI component for the user to choose their own."

**Permissions caveat.** `RegisterEventHotKey` itself does not require Accessibility permission (unlike `CGEvent` taps), but note the documented macOS 15 regression (FB15168205) where sandboxed apps' Option/Option+Shift-only chords stopped firing — disabling sandbox is the current workaround. `NSEvent.addGlobalMonitorForEvents` is an alternative but only observes (cannot consume) events and needs Accessibility.

**Mapping chord→app.** On the hotkey callback, resolve the target app's bundle ID and call `NSWorkspace.openApplication` (activate if running, launch otherwise).

**API design — host-configured vs plugin-registered.** Recommendation for Vee: **the host owns hotkey registration and the recorder UI; plugins declare desired hotkey "actions" in their manifest and the host binds them.** This centralizes conflict detection, lets the user remap, persists bindings in one place, and avoids each plugin process fighting over the single global `RegisterEventHotKey` namespace. Sindre's library explicitly supports this "register a name, let the user record it" model.

## CROSS-CUTTING: SECURITY & STORAGE

**Secret storage in the Keychain.** Store plugin tokens (GitHub PAT, Jira token, etc.) as `kSecClassGenericPassword` items via the Security framework. Core calls: `SecItemAdd`, `SecItemCopyMatching`, `SecItemUpdate`, `SecItemDelete`. Use `kSecAttrService` (e.g., `com.vee.<plugin-id>`) and `kSecAttrAccount` (e.g., the account name) as the composite key, store the token as `kSecValueData`, and set `kSecAttrAccessible` to `kSecAttrAccessibleWhenUnlocked` (or `...ThisDeviceOnly` to prevent iCloud sync). Handle `errSecDuplicateItem` (-25299) by switching to `SecItemUpdate`. **Never give plugins raw Keychain access** — expose a bridged `vee.keychain.get(key)/set(key,value)` that namespaces every item under the calling plugin's id so one plugin cannot read another's secrets. Maintained wrappers like `swift-security` (typed `SecItemQuery<GenericPassword>`, optional biometric `AccessPolicy`) reduce the C boilerplate.

**Capability/permission manifest.** Give each Vee plugin a manifest declaring required capabilities — network domains, filesystem paths (scoped), clipboard, calendar, keychain namespaces, hotkey actions. The host enforces these at the bridge: `vee.http.fetch` rejects domains not in the allowlist; `vee.fs` only resolves paths under declared roots; calendar/clipboard are gated. This is the "configure what an extension is allowed to do" model Raycast describes; they ultimately leaned on open-source review because per-process sandboxing on macOS is hard (`sandbox-exec` is "not supported for third-party development"). For a personal/small-team tool, manifest-enforced capability gating at the bridge is the pragmatic, sufficient layer.

**File protection / encryption for persisted data.** For clipboard history and caches, set `NSFileProtection`/`URLFileProtection` (or `kSecAttrAccessible`-equivalent file flags) on the store, and prefer storing under the app's container. For genuinely sensitive cached content (e.g., clipboard items not marked concealed but still private), encrypt at rest — derive a key stored in the Keychain and encrypt the SQLite store or individual blobs. Always drop `ConcealedType`/`TransientType` content before it ever touches disk (Plugin 1), so encryption is defense-in-depth rather than the only safeguard.

## Recommendations (staged build plan for Vee)

**Stage 1 — Prove the core loop (2–3 weeks).** Build the host process + one `JSContext` per plugin on separate `JSVirtualMachine`s. Bridge `console`, `setTimeout`/`setInterval`, and a `URLSession`-backed `fetch`. Wire esbuild `context().watch()` → FSEvents → context teardown/re-eval for hot reload. Ship a **plain JSON component-tree protocol** (no React yet) with JSON-Patch diffing into AppKit view models. **Benchmark:** end-to-end render-tree update should be <16 ms for typical lists; if not, profile the diff/serialize step before adding React.

**Stage 2 — Latency & data (1–2 weeks).** Implement the native fzy-style matcher and the fetch-once/filter-natively split; the candidate set must never re-cross IPC on keystroke. Add the SWR cache with a pluggable strategy protocol. **Benchmark:** keystroke→filtered-results <5 ms for 10k items; batch+yield if you exceed it.

**Stage 3 — Plugins, security-first (3–4 weeks).** Build App Search and Clipboard History as **host-native providers** first (no network, highest value, prove the privacy model). Then Meeting Bar, API-to-Menubar, PRs/Jiras as JS plugins. Put all tokens in the Keychain behind `vee.keychain` from day one. Add the capability manifest and bridge-level enforcement before any plugin makes a network call.

**Stage 4 — Decide on React.** Only add the `react-reconciler` custom renderer once the JSON protocol is stable and you want richer authoring ergonomics. The wire format (tree → JSON Patch) does not change; React becomes an authoring layer.

**Thresholds that change the plan:**
- If Vee ever needs true npm-package compatibility (OctoKit, etc.), switch the runtime from JSC to a managed Node/Bun child process (Raycast's exact pivot; Bun is JSC-based and TS-native, a natural middle ground).
- If a single plugin needs heavy concurrent compute, give it its own `JSVirtualMachine` on a dedicated thread.
- If memory pressure from per-plugin VMs is high, consolidate low-risk plugins into a shared VM (accepting weaker isolation).

## Caveats

- **Raycast's published architecture is from a May 2023 engineering blog**; details (e.g., worker-thread isolation, gzip thresholds) may have evolved since. Treat it as a proven pattern, not a current spec.
- **JSC's lack of a Node-compatible event loop** means Promise/timer ordering differs from Node and Safari; the Apple Developer Forums thread documents real ordering surprises. Budget time to build a proper microtask/macrotask queue if plugins rely on subtle async ordering.
- **The macOS 15 sandbox + RegisterEventHotKey regression (FB15168205)** for Option-only chords is real; verify your specific chords on your target OS.
- **Jira's `/rest/api/3/search/jql` pagination has documented, unresolved bugs** (`nextPageToken` looping); the legacy `/search` endpoint is gone, so test pagination thoroughly and keep queries tight.
- **GitHub's Search API 30 req/min limit** is easy to blow through with naive polling; the GraphQL 5,000-points/hour path is safer for frequent refresh.
- **EventKit full-access prompt** is unavoidable for reading events on macOS 14+; request it from the main app, not plugin processes, to keep the TCC story clean.
- **`react-reconciler` is explicitly experimental** ("its API is not as stable as that of React… Use it at your own risk") and is versioned independently; pin it.
- Per-process macOS sandboxing for untrusted plugins is genuinely hard (`sandbox-exec` unsupported for third parties); for a personal/small-team tool where you author the plugins this is acceptable, but do not treat Vee's manifest gating as a security boundary against hostile code.
